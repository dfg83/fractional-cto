# Turning Claude Code Conversations into Blog Posts

*I spent a morning designing an AI pipeline to turn my Claude sessions into publishable stories — and the session I used to design it is the first one I wanted to retell.*

---

## The Thing I Do Every Day That Disappears

I spend my entire workday inside Claude Code. Multiple sessions, different projects — some days it's infrastructure architecture, other days it's building plugins, occasionally it's a deep design exploration that takes hours and spans multiple context windows. Some of these sessions are genuinely interesting. Not all of them, but some. A pivotal design decision. A surprising discovery. A tool built to solve a problem that turns out to be more elegant than expected.

And then it's gone. The conversation just sits in `~/.claude/projects/` as a UUID-named file. Nobody will ever read it again, including me. I have over a hundred of these across my projects. Some of them contain stories worth telling, and I'll never know which ones unless I go looking.

One morning, I went looking.

## Inside the File Cabinet

It started with a simple question: where are Claude Code conversations even stored? I asked Claude to find out. Turns out they live as JSONL files — one JSON object per line, one line per message — in a directory structure keyed by project path. My visual identity research session from a few days earlier was sitting there at 14 MB. Five hundred and fifty-four entries. That seemed like a lot of material.

Then Claude measured what was actually in it.

> 0.6% signal. 82 KB of narrative out of 13.8 MB. The user entries are 92% of the file — tool_result blocks contain raw file contents, grep output, screenshots.

Almost nothing in a Claude Code conversation is actually the conversation. The overwhelming majority is machine-to-machine chatter: tool results containing raw file listings, grep output, base64-encoded screenshots, progress events, internal scheduling operations. The human signal — what I said, what Claude said back, the thinking that happened between us — is a sliver buried inside a mountain of operational noise.

That number changed everything about the design. You can't feed 14 MB to an LLM and ask it to find the story. At roughly 3.9 million tokens, you'd blow through the context window before writing a single word. The first stage of any pipeline has to be a script, not an LLM call — strip the noise deterministically, at zero token cost, before Claude ever sees the data.

## Not Every Conversation Is a Story

My first instinct was to build an autonomous pipeline: parse, narrate, polish. Three stages, no human input, conversation goes in, blog post comes out. It took about ten minutes to realize why that wouldn't work.

I asked: "Some conversations make good stories. Some won't. Is it possible to have an interactive mode — after the initial discovery — where I decide which story to tell?"

Claude's response reframed the whole project:

> The current architecture assumes a fully autonomous pipeline — but the best blog posts come from editorial judgment about which thread to pull. A conversation can have multiple possible stories, and only you know which one is worth telling.

A conversation isn't a story. A conversation is raw material that could become several different stories, or no story at all. The same session could be "How I built a visual identity with AI" or "Why I killed the monochrome direction" or "What four parallel AI agents discovered about color palettes." Each angle pulls a different thread, and which thread matters is an editorial decision no model can make for me.

The autonomous pipeline collapsed into something better: five interactive stages with human gates. First, a cheap triage pass — read only the main conversation signal, about 23K tokens, and propose three to five possible story angles with a recommendation. I review, pick an angle, inject any context the conversation doesn't capture, and only then does the expensive work begin. An outline stage structures the post. A draft stage writes it. A polish loop refines it.

The triage stage costs about eight cents on Sonnet. Cheap enough to scan any conversation and find out if there's a story before committing to the full pipeline.

## What the Conversation Can't Know

The JSONL file contains what happened. It doesn't contain why it matters.

Midway through the design session, I realized this gap concretely. I asked: "Would it also be possible to inject custom information? The brainstorming skill that appeared in that conversation is its own story — we built it across a separate session, from research to planning to the final plugin."

That brainstorming skill — a Claude Code plugin I'd built earlier for structured creative thinking — had appeared in the visual identity session as just another tool call. But it wasn't just another tool. It was something I'd designed and built specifically for that kind of work, across its own multi-session arc. The pipeline had no way to know that from the file alone.

Claude put it plainly:

> The agent only sees what's in the JSONL. It has no idea that the brainstorming skill has its own origin story, or that a tool was custom-built, or that you had a specific motivation that wasn't stated in the chat. That's exactly the kind of context that turns a good blog post into a great one.

So the pipeline gained an author context layer. At any interactive gate — after triage, after the outline, during polish — I can inject backstory, motivation, corrections, scope directives. The agent reads the conversation; I tell it what the conversation can't say. The context is tiny, a few hundred tokens at most, but it travels forward through every subsequent stage.

The triage stage now also generates context questions — proactive prompts about moments where something deeper seems to be going on. A custom tool that appeared out of nowhere. A motivation hinted but never stated. A tension that could be a narrative thread if the author confirms it. The pipeline is designed to ask, not to guess.

## Naming It

After three and a half hours of architecture, we needed a name. Claude proposed three: retell, chronicle, recount.

> retell — simple, accurate. You lived it, now you retell it.

That was it. I didn't need to think about the other two.

The last question was whether this should be a skill or a plugin. Claude answered immediately: plugin. The reasoning was clear — this thing needs Python scripts for parsing, multiple stages with different models, a slash command as the entry point, interactive gates between stages, and file I/O. "A skill is just a prompt. This is a full pipeline."

`/retell` it is.

## This Post Is the First One

Here's the part I've been waiting to tell you.

I built retell inside a Claude Code session. Then I ran `/retell` on that session. The parser extracted 165 events. The triage agent — running on Sonnet — read the signal and proposed five story angles, recommending the meta-recursive one: "Building the Tool That Writes My Blog Posts." It spotted the recursion before I named it. I picked the angle, injected author context about my daily workflow and the brainstorming skill's backstory, and the outline agent structured six sections with key quotes and word count targets.

Then the draft stage wrote this post.

The tool's first test subject was its own origin story. The conversation that designed retell became the first conversation retell processed. You're reading the output.

I work on many different problems in Claude Code every day. Some of them are inherently interesting — not all, but some. Until now, those stories evaporated when the session ended. Retell is how I stop that from happening. A conversation goes in, a story comes out, and the human gets to decide which thread to pull.

This was the first thread. There will be more.
